## Task

Integrate constrained OpenCode reasoning. Turn each untrusted transcript into one safe, concise, same-language answer or debate insight through the pinned fast model without shell injection or tool execution.

## What was tried

### 1. Initial implementation

Four new files (OpenCodeTypes, OpenCodeClient, OpenCodeClientFake, OpenCodeTests) with 56 tests. Passed Spec/Code Quality but Security failed at 82. Issues:
- Prompt/transcript passed through argv (visible via `ps`)
- `DispatchGroup.wait()` blocking, not cancellation-aware
- Unbounded stdout/stderr accumulator
- Language not carried from Whisper into prompt
- Static delimiter could be closed by transcript
- No input size bounding

### 2. First correction

All 6 issues fixed: stdin instead of argv, cancellation-aware polling wait loop, bounded accumulator with early termination, language parameter carried through, per-request nonce delimiter with collision retry, input size bounding. 73 tests passed.

Security review passed Spec/Code Quality but failed Security at 85. New issues:
- `--pure` disables external plugins but built-in bash/read/edit tools remained active (global config defaults to allow)
- Timeout test used real opencode with sleep script, not a controlled executable path

### 3. Second correction

Both new issues fixed:
- `OPENCODE_PERMISSION` env var set in child process with deny-all JSON covering every built-in tool (`bash`, `read`, `edit`, `glob`, `grep`, `webfetch`, `task`, `skill`, `lsp`, `question`, `websearch`, `write`, `todowrite`, `external_directory`) plus `"*":"deny"`. Provider credentials preserved from parent environment.
- Live adversarial tests verify: raw NDJSON has zero `tool_use` events with actual execution, no sentinel side-effect file created.
- Controlled executable timeout test using bash wrapper script, plus direct escalation test.

## Why it failed

Three independent review rounds each caught different categories of issues. The fundamental challenge is that OpenCode's permission model defaults to allow for built-in tools, and `--pure` only disables external plugins. Making a single `opencode run` invocation truly zero-tool required discovering and setting `OPENCODE_PERMISSION` environment variable per child process. This environment variable approach works but relies on undocumented OpenCode behavior that could change across versions.

Timeout/cancellation testing proved difficult because `Process` API in Foundation is synchronous and requires bridging to async Swift via polling or continuations. The cancellation-aware polling loop is a pragmatic compromise.

## Suggested next step

1. Monitor OpenCode upstream for a first-class `--no-tools` or `--deny-all` flag on `opencode run` that doesn't require environment variable injection.
2. If `OPENCODE_PERMISSION` format changes, update the `denyAllPermissionJSON` static property accordingly.
3. Consider using `withTaskCancellationHandler` + `withCheckedContinuation` instead of polling if Swift adds a native async Process API in the future.
4. The `processTerminated` error for non-zero exit codes could mask permission denial vs. other errors; add an error classification helper if needed.
