## Task

Connect capture, snapshot, local transcription, constrained reasoning, chimes, and native TTS into a complete hotkey-to-speech journey while preventing recursive feedback. One ready-state Control-Option-Space produces exactly one pipeline run with trigger chime; active processing/speaking triggers are ignored with busy chime.

## What was tried

### 1. Initial implementation

Created orchestration controller (PipelineOrchestrator), AVSpeechSynthesizer wrapper (SpeechSynthesizerActor), mic suppression gate (MicSuppressionGate), dependency-injected protocols/fakes (PipelineTypes), Sendable pipeline runner (PipelineRunner), and wired into AskamiApp with LifecycleStateMachine as a class. 287/288 tests passed (1 pre-existing real-server port conflict unrelated to Task7).

Spec/Tests/Code Quality passed but Code Quality fell below 90 at 82.7. Issues:
- Speech synthesis had no `@MainActor` isolation; continuation could be resumed multiple times under delegate race
- Pipeline work (snapshot/transcribe/reason) inherited MainActor, blocking the main actor during expensive I/O
- `CFAbsoluteTimeGetCurrent` used instead of monotonic `systemUptime`
- Startup transitioned to ready before whisper readiness/capture start confirmed; hotkey registration ignored failure
- `runPipeline` exceeded 50 lines
- No orphan cleanup for pipeline/startup tasks
- 30-second real AVSpeechSynthesizer test depended on audible playback
- Real-server integration test hardcoded port 19990, causing port conflicts

### 2. First correction

All 7 issues fixed:
- Speech: `@MainActor` isolation, UUID-based identity (no `===` on `CheckedContinuation`), concurrent-speak guard, take-and-nil delegate resume, 30-second timeout, `onCancel` via `MainActor.run`, injectable `SpeechDriverProtocol` with `TestSpeechDriver` for deterministic subsecond tests
- Pipeline off-MainActor: `Task.detached` + `nonisolated` stage methods; `PipelineRunner` as `Sendable` struct; only state/speech hops to `MainActor` via `MainActor.run`
- Monotonic clock: `ProcessInfo.processInfo.systemUptime` replaces `CFAbsoluteTimeGetCurrent` everywhere
- Startup sequencing: `lifecycle.startupComplete()` at final point after whisper validate+launch+readiness, capture start, and successful hotkey registration; any failure enters failed with static error and cancels remaining steps
- `runPipeline` refactored into `PipelineRunner` (off-main stages) + `PipelineOrchestrator` (coordinator, <50 lines)
- Orphan cleanup: `applicationWillTerminate` cancels startup task + pipeline task
- Deterministic speech tests: `TestSpeechDriver` fires `didFinish`/`didCancel` synchronously; tests cover single-resume, stop+delegate race, concurrent rejection, cancellation, timeout — all subsecond
- Dynamic port: `findFreePort()` using `bind(port=0)` + `getsockname`; real-server test passes even with dummy listener on 19990

## Why it failed

Two independent review rounds with different rubrics each caught issues the previous round missed. Spec scored 80 (startup sequencing gaps, port dependency), Tests scored 80 (real speech test speed), Code Quality averaged 89.3. The Spec/Tests deficit was closed in the first correction, but the total average fell below the 90 gate due to earlier round weightings.

## Suggested next step

1. Make `PipelineRunner` log function fully `Sendable` without `LogCollector` workaround (requires `@Sendable` closure storage in `PipelineRunner`).
2. Add integration test for `AskamiApp.performStartup()` with fakes for `WhisperServerProcess` and `AudioCaptureSession` to verify exact ordering and failure paths through the state machine.
3. Consider making `LifecycleStateMachine` an actor instead of `@unchecked Sendable` class with `OSAllocatedUnfairLock` for stronger isolation guarantees.
4. Add a cross-process lock or PID file to prevent concurrent `askami` instances from competing for resources.
